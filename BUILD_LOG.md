- 2026-05-18 — v0.2.4: bulletproof stop pipeline. Root causes: (1) `stopRecording()` had no concurrency guard — two rapid stop clicks both passed the `loadSession()` check before either call cleared the session, causing dual `offscreen:stop_audio` messages that hung the first promise forever and left the MediaRecorder running; (2) `waitForAudioDone` timeout resolved the SW promise but never called `closeOffscreen()`, so a hung offscreen kept the mic open indefinitely; (3) the control-bar stop button had no idempotency flag and no visual feedback — `.catch(() => {})` silently dropped errors; (4) no `is_stopping` gate in `onContentEvent`/`captureTabAndQueue`, so events kept flowing during the ~200 ms async window before `saveSession(null)`. Fixes: `stopInFlight` Promise deduplicate all concurrent stop calls (background/index.ts); `state.is_stopping = true` written before any awaits, gating out late events; `closeOffscreen()` called on audio-done timeout; control-bar stop button disabled + shows "…" immediately, retries once, shows "!" error state if both attempts fail; 30-minute max-duration alarm auto-stops and notifies via `chrome.notifications`; SW cold-start now detects a dead offscreen when `mic_enabled` is true and marks the recording `failed` with a notification rather than silently resuming without audio. Added `notifications` permission to manifest. Bumped to v0.2.4. Three new smoke tests cover: single stop clears session, double-stop is idempotent, `is_stopping` gates out content events.
